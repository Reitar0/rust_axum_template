# Дизайн: типовой стартовый шаблон веб-API на Rust

- **Дата:** 2026-07-11
- **Статус:** утверждён (дизайн), ожидает плана реализации
- **Автор:** Asanuri (при участии Claude)

## 1. Цель и контекст

Создать переиспользуемый **стартовый шаблон** для бэкенд-проектов на Rust, с которого
пользователь сможет начинать новые веб-API. Шаблон должен быть «сбалансированным
каркасом»: готов к боевому использованию (config, логи, ошибки, health-check,
миграции, Docker), но не перегружен тем, что зависит от конкретного проекта
(аутентификация, CORS, метрики — вне объёма, добавляются по потребности).

Проект создаётся в существующем репозитории `rust_axum_template` (позже переименуется
в имя реального проекта). Пользователь — новичок в Rust, поэтому код должен быть
идиоматичным и снабжён понятными комментариями, а решения по реализации принимает
разработчик (не выносятся на пользователя как микро-вопросы).

## 2. Ключевые решения (стек)

| Слой | Выбор | Обоснование |
|---|---|---|
| Веб-фреймворк | **axum** | де-факто выбор новых проектов, чистая композиция, от команды tokio |
| Async-рантайм | **tokio** | стандарт экосистемы, база для axum и sqlx |
| Доступ к БД | **sqlx** + **PostgreSQL** | сырой SQL с compile-time проверкой, async, встроенные миграции |
| Архитектура | **Vertical Slice Architecture (VSA)** | группировка по фиче, а не по слою; хорошо масштабируется по мере роста фич |
| Инфраструктура БД | **Docker Compose** | PostgreSQL в контейнере; движок Docker уже установлен (29.6.1, compose v5.2.0) |

## 3. Архитектура — Vertical Slice

Код группируется **по фиче** (вертикальный срез), а не по техническому слою.
Всё, что нужно одной операции — маршрут, обработка, логика, SQL — лежит вместе.
Сквозные вещи вынесены в тонкий `shared/`.

### Структура каталогов

```
<project>/
├── Cargo.toml
├── .env                        # локальные настройки (в .gitignore)
├── .env.example                # образец настроек (в git, без секретов)
├── .dockerignore
├── .gitignore                  # уже есть: /target
├── Dockerfile                  # multi-stage сборка
├── compose.yaml                # PostgreSQL (+ описание сервиса app)
├── README.md                   # шпаргалка команд
├── migrations/                 # SQL-миграции sqlx
│   └── 0001_init.sql
└── src/
    ├── main.rs                 # тонкая точка входа: init → запуск
    ├── app.rs                  # сборка Router из всех фич + middleware
    ├── shared/                 # сквозное, общее для всех срезов
    │   ├── mod.rs
    │   ├── config.rs           # чтение конфигурации из окружения
    │   ├── error.rs            # AppError + impl IntoResponse
    │   ├── state.rs            # AppState { db pool, config }
    │   └── db.rs               # создание пула, запуск миграций
    └── features/               # вертикальные срезы
        ├── mod.rs
        └── health/
            ├── mod.rs          # pub fn router() -> Router<AppState>
            └── handler.rs      # GET /health (+ проверка живости БД)
```

### Принципы
- `main.rs` **не содержит бизнес-логики** — только «собрать приложение и запустить».
- Каждый feature-модуль экспортирует свой `Router`, корневой `app.rs` собирает их
  через `.merge()` / `.nest()`.
- `shared/` остаётся тонким: туда попадает только действительно сквозное
  (конфиг, тип ошибки, пул БД, AppState).
- Когда какой-то срез разрастётся — он локально рефакторится, не затрагивая другие.

## 4. Зависимости

Версии в спеке не фиксируются — при реализации добавляются через `cargo add`
(установленный `cargo-edit` подтянет последние совместимые версии). Точные
сигнатуры API axum/sqlx сверяются с актуальной документацией (context7) на этапе кода.

| Крейт | Роль | Features (ориентир) |
|---|---|---|
| `axum` | веб-фреймворк | — |
| `tokio` | async-рантайм | `rt-multi-thread`, `macros`, `signal`, `net` |
| `tower-http` | middleware | `trace`, `timeout`, `request-id` |
| `sqlx` | доступ к БД | `runtime-tokio`, `tls-rustls`, `postgres`, `macros`, `migrate`, `uuid`, `chrono` |
| `serde` / `serde_json` | (де)сериализация | `serde` с `derive` |
| `tracing` / `tracing-subscriber` | логирование | `env-filter` |
| `thiserror` | тип ошибок приложения | — |
| `anyhow` | ошибки старта/`main` | — |
| `config` | чтение конфигурации | — |
| `dotenvy` | загрузка `.env` локально | — |
| `uuid`, `chrono` | идентификаторы и время | согласованы с features sqlx |
| *(dev)* `reqwest` | интеграционные тесты | — |

**Заметки по выбору:**
- `tls-rustls` (чистый Rust), а не `native-tls` — не тянет системный OpenSSL,
  упрощает Docker-сборку.
- `config` читает из окружения в типизированную структуру; `dotenvy` только
  локально подгружает `.env` в окружение. В проде `.env` нет — настройки из
  окружения контейнера, код не меняется.

## 5. Поток запроса

```
HTTP-запрос
  → tower-http middleware (TraceLayer, TimeoutLayer, RequestId)
  → axum Router (app.rs): сопоставление метод+путь со срезом
  → handler фичи: extractors (State, Path, Query, Json) достают нужное
  → логика среза + SQL (sqlx); ошибки пробрасываются через `?`
  → Result<T, AppError>
      Ok(T)         → Json(T)
      Err(AppError) → AppError::into_response() → HTTP-статус + JSON
  → HTTP-ответ
```

## 6. Обработка ошибок

Единый тип `AppError` (на `thiserror`), реализующий `IntoResponse`. Хендлеры
возвращают `Result<T, AppError>`; оператор `?` пробрасывает ошибки, маппинг в
HTTP происходит централизованно.

Набор вариантов на старте:

| Вариант | HTTP | Назначение |
|---|---|---|
| `NotFound(String)` | 404 | сущность не найдена |
| `Validation(String)` | 422 | некорректные входные данные |
| `Unauthorized` | 401 | задел под будущую аутентификацию |
| `Database(sqlx::Error)` | 500 | ошибка БД (деталь — в лог, наружу — общее сообщение) |
| `Internal(anyhow::Error)` | 500 | прочие внутренние ошибки |

**Принципы:**
- `#[from]` в `thiserror` генерирует `From<sqlx::Error>` и `From<anyhow::Error>`,
  чтобы `?` работал бесшовно.
- Клиенту — безопасное сообщение (`{"error": "..."}`), полная причина пишется в
  `tracing`. Детали БД не протекают в API.

## 7. Конфигурация

```
struct Config {
    server_addr: SocketAddr,   // APP_SERVER_ADDR, дефолт 0.0.0.0:8080
    database_url: String,      // DATABASE_URL (обязателен; при отсутствии — понятная ошибка старта)
    log_level: String,         // APP_LOG_LEVEL / RUST_LOG, дефолт "info"
}
```

- Читается один раз при старте (`config`, префикс `APP_`); локально — из `.env`
  через `dotenvy`.
- `Config` живёт внутри `AppState` вместе с пулом БД.
- `.env.example` документирует все переменные; `.env` — в `.gitignore`.

## 8. База данных и миграции

- Пул соединений (`sqlx::PgPool`) создаётся при старте в `shared/db.rs`.
- Миграции — SQL-файлы в `migrations/`, применяются автоматически при старте
  через `sqlx::migrate!()`. База помнит применённые, накатывает только новые.
- `0001_init.sql` на старте может быть минимальным (например, комментарий-заготовка
  или простая служебная таблица) — реальные таблицы добавятся с фичами.

## 9. Docker и инфраструктура

- **`compose.yaml`** — сервис `db` (PostgreSQL заданной версии, проброс порта,
  volume для данных, healthcheck). Опционально описан сервис `app` для упаковки,
  но на этапе разработки сервис запускается локально через `cargo run` (быстрее
  итерации), а в Docker крутится только база.
- **`Dockerfile`** — multi-stage: стадия сборки на официальном rust-образе,
  финальная стадия — тонкий образ (debian-slim) с одним бинарником и `tls-rustls`
  (без OpenSSL). Итоговый образ компактный.
- **`.dockerignore`** — исключает `target/`, `.env`, `.git` из контекста сборки.

## 10. Тестирование

- **Интеграционный тест health:** поднять приложение, запросить `GET /health`
  через `reqwest`, проверить 200 и статус живости БД.
- **Юнит-тесты** — там, где появляется логика внутри срезов.
- Прогон через **`cargo nextest run`** (установлен).
- Тесты, требующие БД, используют подключение к PostgreSQL из окружения
  (compose поднимает базу перед прогоном).

## 11. Удобство работы (DX)

Шпаргалка команд в `README.md`:

| Команда | Что делает |
|---|---|
| `docker compose up -d` | поднять PostgreSQL |
| `cargo run` | запустить сервис (миграции накатятся сами) |
| `cargo watch -x check` | автопроверка кода при сохранении |
| `cargo nextest run` | прогнать тесты |
| `cargo deny check` | проверка зависимостей (безопасность/лицензии) |
| `docker compose down` | остановить и убрать базу |

## 12. Definition of Done

Шаблон считается готовым, когда:
1. `docker compose up -d` поднимает PostgreSQL.
2. `cargo run` стартует сервер, автоматически применяя миграции.
3. `GET /health` возвращает 200 и статус подключения к БД.
4. При отсутствии `DATABASE_URL` — понятная ошибка старта (не паника без контекста).
5. `cargo nextest run` — зелёный (health-тест проходит при поднятой базе).
6. `.env.example` описывает все переменные окружения.
7. `README.md` содержит рабочую шпаргалку команд.
8. Структура каталогов соответствует VSA из раздела 3.

## 13. Вне объёма (на будущее)

Сознательно **не** входит в стартовый шаблон, добавляется по потребности проекта:
- Аутентификация/авторизация (JWT) — но `AppError::Unauthorized` уже заложен.
- CORS, rate-limiting.
- Метрики (Prometheus), OpenAPI-документация.
- Пример CRUD-ресурса (шаблон ограничивается срезом `health`).
